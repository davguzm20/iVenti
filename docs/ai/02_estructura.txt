FUNCIONES DE CADA CAPA
======================

ENTITIES (features/[feature]/entities/)
----------------------------------------
Propósito: Representar objetos de negocio puros
Reglas:
  - NO imports de database (sqflite, postgres, etc.)
  - NO métodos static Future (asincrónicos)
  - SOLO propiedades, constructores, toMap(), fromMap(), copyWith()
  - Inmutables (usar final en propiedades)
Ejemplo: Producto, Cliente, Venta

DTOS (features/[feature]/dtos/)
-------------------------------
Propósito: Transferir datos entre capas
Tipos:
  - requests/: Para entrada de datos (con validación)
  - responses/: Para salida de datos (solo datos)
Ejemplo: CrearProductoRequest, ProductoResponse

REPOSITORIES (features/[feature]/repositories/)
-----------------------------------------------
Propósito: Acceso a datos + validaciones
Responsabilidades:
  - TODO el SQL (INSERT, UPDATE, DELETE, SELECT)
  - Validaciones de formato e integridad
  - Conexión a PostgreSQL via database_config
Ejemplo: ProductoRepository.crearProducto()

SERVICES (features/[feature]/services/)
---------------------------------------
Propósito: Lógica de negocio y reglas
Responsabilidades:
  - Reglas de negocio complejas
  - Coordinar múltiples repositories
  - Auditoría y trazabilidad
  - Excepciones de negocio
Ejemplo: ProductoService.validarYCrearProducto()

CONTROLLERS (features/[feature]/controllers/)
---------------------------------------------
Propósito: Gestión de estado de UI
Responsabilidades:
  - Extender ChangeNotifier (Provider)
  - Manejar loading, error, data states
  - Conectar UI con Services
  - Notificar cambios a UI
Ejemplo: InventoryController.cargarProductos()

PAGES (features/[feature]/pages/)
---------------------------------
Propósito: UI pura
Reglas:
  - NO lógica de negocio
  - NO imports de repositories o services
  - SOLO widgets y estado visual
  - Usar controllers para estado
Ejemplo: InventoryPage, CreateProductPage


FLUJO DE TRABAJO TÍPICO
=======================

1. Usuario ingresa datos en Page
2. Page llama a Controller
3. Controller valida y llama a Service
4. Service aplica reglas de negocio
5. Service llama a Repository
6. Repository valida y ejecuta SQL
7. PostgreSQL guarda datos
8. Response sube por las capas
9. Controller actualiza estado
10. Page muestra resultado
